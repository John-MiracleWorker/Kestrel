# Implementation Status

Last updated: 2026-07-19

This repository is a production candidate for the supported single-user, single-node local/private profile. Hosted/team identity, multi-tenant isolation, distributed scheduling, and unsupported extension surfaces remain outside that profile. The status below is intentionally literal so future Codex passes can harden those broader surfaces without weakening the supported contract.

- Persistent recall trust boundary: packed memory and failure lessons are JSON-encoded as untrusted user-role evidence, so content copied from tools, web pages, files, or channels is never injected as a system-priority instruction. Soul system content is limited to the fixed persona preset selected through the authenticated onboarding route; display labels and free-form Soul preferences are bounded untrusted JSON at user-role priority. Policy recall receives system priority only when `memory.policy_promote` metadata cross-checks against the durable owner-approved exact call and its recorded result. Activated behavior deltas remain separate trusted runtime messages, and deterministic capability/approval gates remain authoritative.

## Current Contract

- Recent transcript replay is limited to 8 completed real primary/channel user turns and 8,000 characters. The current user frame is excluded from both transcript replay and recalled-memory hits; internal scheduler, subagent, and approval-continuation turns carry `transcript_scope=internal` and never replay as native user authority.
- Recalled working, episodic, tool, web, file, and channel content is untrusted evidence at user-message priority, even after it persists across turns. It cannot become a system instruction through memory retrieval.
- Free-form Soul fields (`working_style`, goals, interests, and communication notes) stay in bounded, explicitly untrusted user-role JSON. Caller-asserted `self.remember` validation/source strings cannot authenticate onboarding or enter the system role.
- `memory.learn` and `memory.consolidate` cannot promote policy. The dedicated `memory.policy_promote` tool uses two separately approved phases: stage an exact episodic proposal, then promote only current-run receipts bound to that proposal and emitted by the correct validation tool class. It requires structured human-explicit validation and at least five distinct task receipt refs; raw signed artifact replay and cross-bucket substitution fail closed. System-priority recall additionally verifies the durable final approval row, exact argument digest, recorded result ID, proposal, receipts, and any signed repair/review backing.
- `memory.import` strips runtime transcript-authority fields and stamps imported records as untrusted. Approval authorizes the data write; it does not authenticate imported content as a native user/assistant transcript or validated policy.
- Transcript replay requires the current turn's exact primary/channel scope and origin. Existing lossless channel session IDs remain stable; collision-prone normalized or truncated identifiers receive a versioned digest.
- Onboarding and policy candidates are scanned and authenticated before bounded selection, so untrusted records cannot crowd a trusted persona or durable policy constraint out of system context.
- A criterion claiming tests, lint, checks, or validation passed requires trusted validation-producing evidence. Generic successful tools remain useful task evidence but cannot prove a validation outcome.
- High-risk tools require both capability enablement (matching allow flag, where applicable) and explicit approval for the exact tool-call ID and arguments before execution.
- Capability switches express owner-desired state; effective state is the conjunction of that switch, immutable master/config prerequisites, launch allowlists, effective parent/plugin state, and unchanged resource digests. Enabling never bypasses an exact-call approval or master flag.
- Capability changes apply to future invocation attempts. Turning off is rechecked by stale registries and active runs before later dispatch, but it does not promise to cancel an arbitrary built-in process that already started. Turning on does not rewrite an active run's snapshotted configuration.
- Pending approvals for a disabled tool or child of a disabled skill/MCP server are denied. Approved continuations are valid only while the exact-call identity, capability revision, tool policy/specification digest, and parent resource digest still match.
- `NEST_AGENT_ALLOW_WEB` / `--allow-web` enables read-only web tools; web fetches reject private, local, link-local, multicast, reserved, and unspecified addresses.
- `NEST_AGENT_ALLOW_SELF_MODIFICATION` / `--allow-self-modification` only enables the high-risk `self.propose_change` request path. It does not bypass exact-call approval or the repair/commit gates.
- The agentic failure cycle is default-on through `enable_agentic_cycle`; disable it only for debugging with `NEST_AGENT_DISABLE_AGENTIC_CYCLE=1`.
- Autonomous scheduling is opt-in and bounded by `max_scheduler_tasks` / `NEST_AGENT_MAX_SCHEDULER_TASKS` per cycle and `max_scheduler_cycles` / `NEST_AGENT_MAX_SCHEDULER_CYCLES` per drain run.
- Proactive routine polling is separately opt-in through `enable_proactive_routines` / `NEST_AGENT_ENABLE_PROACTIVE_ROUTINES`; owner mutations through the API require the shared API-auth gate even on loopback. New routines start disabled, intervals are at least 60 seconds, and each tick is bounded by `max_routines_per_tick` / `NEST_AGENT_MAX_ROUTINES_PER_TICK`.
- Scheduled routine prompts and responses use `turn_origin=scheduled_routine` and `transcript_scope=internal`, so they do not replay later as native primary/channel user authority. A running or approval-blocked occurrence counts as active for overlap suppression.
- Automatic memory compaction is opt-in through `NEST_AGENT_ENABLE_AUTO_COMPACT=1` and dry-run unless `NEST_AGENT_AUTO_COMPACT_APPLY=1`.
- Git worktree isolation for scheduler/subagent execution is opt-in through `enable_worker_isolation` / `NEST_AGENT_ENABLE_WORKER_ISOLATION`; isolated worker branches use `worker_branch_prefix` and `worker_worktree_dir`.
- Cancelled runs must not transition to completed, blocked, or failed after cancellation; lifecycle updates should use the guarded state transition helper.
- Completed, failed, and cancelled runs are immutable even for repeated same-status transition attempts; approval requests are immutable after leaving `pending`.
- Tool execution is bounded by `tool_timeout_seconds` / `NEST_AGENT_TOOL_TIMEOUT_SECONDS` and timeout failures are returned as structured tool errors. Transactional repair/commit tools request cancellation at that boundary and wait for their bounded worker to stop before returning, so a reported terminal timeout cannot leave a background mutation running.
- Behavior-delta runtime/preflight compilation is default-off through `NEST_AGENT_ENABLE_BEHAVIOR_DELTAS=0`. When enabled, it only compiles active relevant deltas, logs activations, and remains advisory. Proposed/staged low-risk deltas can auto-activate only when `NEST_AGENT_ENABLE_AUTO_ACTIVATE_LOW_RISK_DELTAS=1` and explicit validation metadata plus `MutationGate` evidence thresholds pass; policy/high-risk gates are unchanged, and broad automatic blocking is not performed beyond existing deterministic retry blocking.
- Non-secret runtime settings persist through a revision-checked transaction shared by the workbench and Telegram admin path. Candidate paths are validated before persistence; stale writers fail with HTTP 409; activation failure restores both the previous owner-only file and live config; API authentication remains launch-controlled. The Settings UI exposes `max_tool_rounds` as “Max tool calls” for new runs.
- New background runs persist a root task plus a small starter DAG with dependencies, required tools, risk, acceptance criteria, attempt count, failure reason, diagnosis, retry-strategy fields, and graph-runtime plan metadata.
- Ordinary conversation and observations must not write policy memory directly or self-assert promotion/approval provenance.
- The Memvid backend must use `.mv2` files and preserve one file per memory layer, including `.nest/memory/self.mv2`.
- `complete.mv2` is a run artifact under `.nest/runs/{run_id}/`, not a permanent memory layer.
- SQLite is control-plane state only. It is not the retrieval memory substrate.
- Support bundle exports are diagnostic artifacts only: they report environment-variable presence, redact common secret patterns, summarize state tables, and do not include raw broker vault contents or `.mv2` memory files.
- Mock-provider tests are the default fast validation path; Memvid integration remains behind `RUN_MEMVID_INTEGRATION=1`.
- Provider fallback only runs for `ProviderError(retryable=True)` failures; non-retryable errors fail fast and preserve the original provider error.
- MCP static tools remain approval-by-default unless a server is explicitly configured to trust reviewed manifest metadata; dangerous tool names/descriptions such as file writes, deletes, shell execution, patching, committing, or secrets are promoted to high risk during vetting. Dynamically discovered live tools always remain at least medium risk and approval-required.
- Invalid skill manifests are rejected during discovery; accepted skill manifests record validation status plus manifest/SKILL.md content hashes for provenance.
- Memory promotion decisions must carry auditable gate metadata, validation evidence refs, provenance, confidence, and validation status; one-off procedural successes and ordinary events can explain why they were not promoted into procedural/policy memory.
- Plugin review metadata can block enablement; it is not a package installer. Executable plugin skills still need an explicitly declared, digest-pinned OCI runtime and canonical scopes before the separate containment runner can launch them.
